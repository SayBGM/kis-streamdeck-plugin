# WebSocket 구독 복구와 시세 카드 상태 텍스트 정리 설계

## 배경

2.0.1 플러그인은 KIS WebSocket 연결 자체는 열지만 구독 성공 응답을 현재 제어 작업과 연결하지 못한다. KIS의 실제 구독 응답은 종목 키를 `header.tr_key`에 담지만, `SubscriptionSupervisor`는 `body.output.tr_key`와 `body.input.tr_key`만 읽는다. 그 결과 정상 응답도 키 없는 응답처럼 폐기되고, 5초 제어 타임아웃마다 열린 소켓을 종료한 뒤 지수 백오프로 재연결한다. 사용자는 이 동안 REST 스냅샷만 표시되는 `BACKUP` 상태를 보게 된다.

현재 일반 시세 카드는 종목명과 연결 점, 종목코드, 장 상태, 주가, 등락률 아래에 `실시간`, `백업`, `시세 지연`, `새로고침 중`, `연결 확인 필요` 같은 상태 텍스트를 추가로 표시한다. 연결 상태는 이미 종목명 색상과 점으로 구분할 수 있으므로 하단 상태 텍스트만 제거하고 기존 UI는 유지한다.

WebSocket은 틱을 매우 자주 전달할 수 있다. 원문이나 시세 샘플을 중간에서 버리면 마지막 수신 시각과 stale 판정이 실제 연결 상태와 달라질 수 있으므로, 모든 시세를 처리한 뒤 화면 반영 경계에서 최신 값만 선택한다.

## 목표

- 실제 KIS 구독 성공 응답의 `header.tr_key`를 정확히 인식해 WebSocket 구독을 `live`로 전환한다.
- WebSocket 시세는 전부 수신하고 상태에는 모두 반영하되, 화면에는 설정된 간격마다 가장 최신 값만 반영한다.
- 일반 시세 카드에서는 하단 상태 텍스트만 제거한다.
- 상태 텍스트가 사라진 공간에 맞춰 주가와 등락률의 세로 간격을 소폭 조정한다.
- 기존 연결 점, 종목명 상태색, 종목코드, 세션 배지, 대기·오류·회복 화면은 유지한다.
- 기존 REST 전송 제한, 연결 백오프, 공개 액션 설정 타입과 시세 수집 정책은 유지한다.

## 비목표

- WebSocket 원문 수신, 파싱 또는 구독 데이터 전달 자체를 샘플링하지 않는다.
- REST의 초당 10회 및 동시 4회 전송 제한을 변경하지 않는다.
- 대기, 오류, 회복 전용 카드를 새 레이아웃으로 다시 설계하지 않는다.
- 연결 점이나 종목명 색상 기반 상태 표현을 제거하지 않는다.
- KIS 서버가 종목 키를 전혀 돌려주지 않는 응답을 현재 제어 작업의 성공으로 추정하지 않는다.

## 검토한 스로틀링 접근

### 1. 기존 렌더 스케줄러를 화면 스로틀 경계로 사용

`RenderScheduler`는 버튼별 마지막값 우선 병합, 의미 키 중복 제거, 직렬 `setImage()` 호출과 2·5·10초 간격 설정을 이미 제공한다. 컨트롤러는 모든 WebSocket 시세를 먼저 상태에 반영하고 완성된 화면 의도만 스케줄러에 제출한다. 따라서 데이터 신선도는 보존하면서 Stream Deck IPC만 제한할 수 있다. 사용자가 승인한 방식이며 이를 채택한다.

### 2. 액션 컨트롤러에서 시세 이벤트를 스로틀

렌더 의도 생성량은 줄지만 버려진 시세가 마지막 수신 시각, 현재가와 stale 전환에 미치는 영향을 별도로 보정해야 한다. 데이터 정책과 표시 정책이 섞이므로 채택하지 않는다.

### 3. WebSocket 원문 메시지를 스로틀

처리량은 가장 많이 줄지만 구독 제어 응답, PINGPONG, 실제 시세를 같은 전송에서 선별해야 한다. 연결 생존성과 프로토콜 정확성을 훼손할 수 있어 채택하지 않는다.

## WebSocket 구독 응답 처리

`SubscriptionSupervisor.handleControlMessage()`는 다음 순서로 응답을 해석한다.

1. `header.tr_id`와 현재 제어 작업의 `trId`가 정확히 일치해야 한다.
2. 종목 키는 KIS 실제 형식인 `header.tr_key`에서 먼저 읽는다.
3. 기존 호환 응답을 위해 `body.output.tr_key`, `body.input.tr_key`를 차례로 허용한다.
4. 읽은 키가 현재 제어 작업의 `trKey`와 정확히 일치해야 한다.
5. 성공 코드 `OPSP0000` 또는 `OPSP0002`일 때만 구독을 `live`로 전환한다.
6. 키가 없거나 다른 키인 응답은 늦게 도착한 이전 응답일 수 있으므로 무시하고 기존 제어 타임아웃을 유지한다.

이 변경은 응답 상관관계의 엄격함을 낮추지 않는다. 키의 공식 위치만 추가로 인식한다. 성공 후 다음 구독은 기존 100ms 제어 간격을 지키며 전송된다. 연결 타임아웃, heartbeat, 지수 백오프는 변경하지 않는다.

## 화면 반영 스로틀링

WebSocket 시세 데이터 흐름은 다음과 같다.

```text
KIS WebSocket 틱
  → SubscriptionSupervisor가 모든 틱 파싱
  → StockActionController가 quote·receivedAt·stale 상태 갱신
  → RenderScheduler에 완성된 화면 의도 제출
  → 설정 간격 동안 같은 버튼의 의도를 마지막값 우선 병합
  → 가장 최신 SVG만 setImage()
```

`normal` 시세 화면은 기존 전역 설정 `renderIntervalMs`의 2초, 5초, 10초를 사용한다. 수동 새로고침과 종목 설정 오류처럼 즉시 확인해야 하는 `immediate` 작업은 기존대로 기다리지 않는다. 연결 상태 변화와 같은 `control` 작업은 기존 1초 제한을 유지한다.

Property Inspector의 `렌더 간격` 표기는 `화면 갱신 제한`으로 바꾼다. 선택값과 저장 형식은 유지해 마이그레이션이나 공개 프로토콜 변경을 만들지 않는다. 진단의 `submitted`, `coalesced`, `renders`, `commits`로 스로틀 작동을 계속 확인할 수 있다.

## UI 변경

일반 시세 카드의 기존 구조를 유지한다.

```text
┌──────────────────┐
│ ● 종목명    장 상태 │
│ 종목코드           │
│                  │
│       주가         │
│      등락률         │
│                  │
└──────────────────┘
```

- 종목명과 연결 점: 기존 `LIVE`, `BACKUP`, `BROKEN`, `waiting` 색상 규칙을 유지한다.
- 종목코드: 기존 위치와 형식을 유지한다.
- 장 상태: 기존 세션 pill과 색상을 유지한다.
- 주가와 등락률: 기존 형식과 상승·하락 색상을 유지하되, 하단 상태 텍스트가 사라진 공간을 고려해 시각적 중심을 소폭 조정한다.
- 하단 상태 텍스트: 일반 시세 카드와 레거시 `renderStockCard`에서 생성하지 않는다.
- `refreshing`일 때 상태 텍스트와 함께 표시하던 로딩 표시도 일반 시세 카드에서는 제거한다. 텍스트 없이 스피너만 남기지 않는다.

대기 카드의 로딩 표시와 안내, 오류 카드, 회복 카드는 기존 동작을 유지한다. 입력 검증 실패와 설정 필요 카드도 변경하지 않는다. 이번 UI 변경은 정상 가격 데이터가 있는 시세 카드에만 적용한다.

`stale`, `refreshing`, `BROKEN`은 화면 모델과 의미 키에 계속 포함된다. 따라서 상태 정책과 렌더 스케줄링은 바뀌지 않지만 일반 시세 카드에 별도 하단 문구를 추가하지 않는다. 연결 상태는 종목명과 점의 색상으로 확인한다.

## 테스트 전략

모든 동작 변경은 실패 테스트를 먼저 확인한 뒤 최소 구현으로 통과시킨다.

### WebSocket

- 실제 KIS 형식인 `header: { tr_id, tr_key }`, `body: { rt_cd, msg_cd, msg1 }` 성공 응답이 현재 구독을 `live`로 만드는 테스트를 추가한다.
- `header.tr_key` 성공 후 제어 타임아웃이 재연결을 일으키지 않는지 검증한다.
- 다른 `header.tr_key`와 키 없는 응답은 현재 작업을 승인하지 않는 기존 보안 테스트를 유지한다.
- `body.output/input.tr_key` 호환 응답 테스트를 유지한다.

### 스로틀링

- WebSocket 시세가 설정 간격보다 자주 들어와도 모든 데이터 이벤트가 처리되고 렌더 커밋은 간격당 한 번만 발생하는 통합 테스트를 추가한다.
- 병합 구간의 마지막 시세가 실제 화면에 반영되는지 검증한다.
- 수동 새로고침과 설정 오류의 즉시 렌더 동작을 유지한다.
- Property Inspector에 `화면 갱신 제한` 표기와 2·5·10초 값이 보이는지 검증한다.

### 카드 UI

- 일반 `StockActionView` 시세 카드에서 `실시간`, `백업`, `연결 확인 필요`, `시세 지연`, `새로고침 중` 상태 텍스트가 생성되지 않는지 검증한다.
- 일반 시세 카드에서 `data-role="loading-indicator"`가 생성되지 않는지 검증한다.
- 종목명과 연결 점, 종목코드, 세션 pill, 주가, 등락률은 계속 생성되는지 검증한다.
- `LIVE`, `BACKUP`, `BROKEN`, `waiting`별 종목명과 점의 색상을 유지하는지 검증한다.
- 레거시 `renderStockCard`도 하단 상태 문구와 로딩 표시를 제거하고 나머지 시세 요소를 유지하는지 검증한다.
- 대기, 오류, 회복, 설정 카드의 기존 의미와 표시가 유지되는지 검증한다.

## 완료 조건

- 2.0.1에서 재현된 5초 구독 타임아웃 재연결 루프가 실제 KIS `header.tr_key` 응답에서 발생하지 않는다.
- WebSocket 구독이 성공 응답 후 `live`로 전환된다.
- WebSocket 시세 이벤트는 모두 처리되며 화면은 설정된 2·5·10초 간격마다 최신 값만 반영한다.
- 일반 시세 카드에서 하단 상태 텍스트와 그 전용 로딩 표시가 사라진다.
- 종목명과 점의 상태색, 종목코드, 장 상태, 주가, 등락률 및 예외 화면은 유지된다.
- REST 제한, WebSocket heartbeat·백오프, 시세 파싱, 공개 설정 저장 형식은 변경되지 않는다.
- 관련 단위·통합 테스트, 타입 검사, 전체 테스트, 커버리지 검증, 프로덕션 빌드와 패키지 smoke 검증이 모두 통과한다.
